# 标签颜色优化产品规格说明书
# Version: 1.0, Created: 2026-03-07

## 1. 项目概述

本项目旨在优化 Chrome 页面管理插件中的标签颜色生成逻辑，从基于预设颜色列表的简单哈希选择，升级为基于 HSL/HSV 色彩空间的动态颜色生成系统，为每个标签名称生成唯一、和谐且高对比度的颜色。

### 1.1 背景与目标

当前实现使用简单的字符哈希算法从 6 种预设颜色中选择，存在以下问题：
- 颜色选项有限，容易重复
- 无法保证不同标签名称生成的颜色有足够差异
- 没有考虑颜色的可读性和对比度

新系统的目标：
- 基于标签名称生成唯一的、可复现的颜色
- 使用 HSL/HSV 色彩空间确保颜色和谐
- 保证文字在标签颜色上的高对比度可读性
- 足够的颜色差异化，避免视觉混淆

## 2. 用户故事

### 2.1 主要用户故事

**作为用户，我希望每个标签都有独特的颜色，这样我可以快速通过颜色识别不同的标签。**

**验收标准：**
- 相同的标签名称始终生成相同的颜色
- 不同的标签名称生成的颜色有明显差异
- 颜色变化基于标签名称的内容，而非随机

---

**作为用户，我希望标签颜色和谐美观，这样页面看起来更舒适。**

**验收标准：**
- 避免使用过于刺眼的高饱和度颜色
- 颜色在 HSL/HSV 空间中分布合理
- 整体视觉效果协调统一

---

**作为用户，我希望标签文字清晰可读，这样我不需要费力就能看清标签名称。**

**验收标准：**
- 标签文字与背景色有足够的对比度
- 深色背景使用浅色文字，浅色背景使用深色文字
- 符合 WCAG 可访问性标准（对比度至少 4.5:1）

## 3. 功能性需求

### 3.1 颜色生成算法

| ID | 需求描述 | 优先级 |
|----|----------|--------|
| FR-1.1 | 基于标签名称的字符串哈希生成 HSL 颜色参数 | P0 |
| FR-1.2 | 色相 (Hue) 范围：0-360 度，均匀分布 | P0 |
| FR-1.3 | 饱和度 (Saturation) 范围：适中（建议 55-75%），避免过于刺眼 | P0 |
| FR-1.4 | 亮度 (Lightness) 范围：适中（建议 45-65%），确保文字可读性 | P0 |
| FR-1.5 | 相同的标签名称必须生成完全相同的颜色 | P0 |
| FR-1.6 | 将 HSL 颜色转换为标准的 hex 格式（#RRGGBB） | P0 |

### 3.2 对比度与可读性

| ID | 需求描述 | 优先级 |
|----|----------|--------|
| FR-2.1 | 自动计算标签颜色的相对亮度 | P0 |

| FR-2.3 | 确保文字与背景色的对比度至少达到 4.5:1 | P0 |
| FR-2.4 | 如果对比度不足，自动调整背景色的亮度以满足要求 | P1 |

### 3.3 现有标签迁移

| ID | 需求描述 | 优先级 |
|----|----------|--------|
| FR-3.1 | 为所有现有标签重新生成颜色 | P0 |

| FR-3.3 | 迁移过程中保留标签的其他属性（名称、ID 等） | P0 |
| FR-3.4 | 迁移完成后立即保存到存储 | P0 |

### 3.4 新标签创建

| ID | 需求描述 | 优先级 |
|----|----------|--------|
| FR-4.1 | 创建新标签时自动使用新算法生成颜色 | P0 |

| FR-4.3 | 编辑标签时，如果未手动选择颜色，保持基于名称的动态颜色 | P0 |

## 7. 边缘场景与异常处理

### 7.1 边缘场景

| 场景 | 处理方式 |
|------|----------|
| 空标签名称 | 使用默认颜色，同时提示用户输入名称 |
| 极短标签名称（1-2 字符） | 确保哈希算法仍然能生成稳定的颜色 |
| 极长标签名称 | 截取合理长度进行哈希，避免性能问题 |
| 特殊字符标签名称 | 正确处理 Unicode 字符，确保哈希稳定 |
| 大小写不同的相同名称 | 统一转换为小写进行哈希，确保"工作"和"工作"生成相同颜色 |

### 7.2 异常处理

| 异常 | 处理方式 |
|------|----------|
| 哈希计算失败 | 回退到预设的默认颜色 |
| HSL 转换失败 | 使用安全的默认颜色值 |
| 存储迁移失败 | 保留现有颜色，记录错误日志 |
| 对比度计算失败 | 保守选择白色或黑色文字 |
