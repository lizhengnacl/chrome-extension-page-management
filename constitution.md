# Chrome 页面管理插件开发宪法
# Version: 1.0, Ratified: 2026-03-04

本文件定义了本项目不可动摇的核心开发原则。所有AI Agent在进行技术规划和代码实现时，必须无条件遵循。

---

## 第一条：简单性原则 (Simplicity First)

**核心：** 遵循"少即是多"哲学。绝不进行不必要的抽象，绝不引入非必需的依赖。

- **1.1 (YAGNI):** 你不需要它（You Ain't Gonna Need It）。只实现明确要求的功能。
- **1.2 (原生优先):** 除非有极其充分的理由，否则必须优先使用原生 JavaScript 和 Chrome API。不使用 React、Vue 等框架。
- **1.3 (反过度工程):** 避免复杂的设计模式。简单的函数和数据结构优于复杂的抽象。

---

## 第二条：用户体验铁律 (User Experience Imperative) - 不可协商

**核心：** 用户体验是插件的灵魂。所有功能都必须从用户角度出发设计。

- **2.1 (即时反馈):** 所有用户操作都必须有即时的视觉反馈（成功提示、加载状态、错误提示）。
- **2.2 (快速访问):** 插件必须快速加载，弹出窗口响应时间不超过 100ms。
- **2.3 (直观操作):** 界面设计遵循直觉，用户无需阅读文档即可使用。

---

## 第三条：明确性原则 (Clarity and Explicitness)

**核心：** 代码的首要目的是让人类易于理解，其次才是让机器执行。

- **3.1 (错误处理):** **不可协商**：所有异步操作都必须使用 try-catch 显式处理。绝不允许静默失败。错误提示必须友好易懂。
- **3.2 (状态管理):** 使用简单的状态对象管理应用状态，状态变更必须清晰可追踪。
- **3.3 (注释的意义):** 注释应该解释"为什么"，而不是"是什么"。关键逻辑必须有清晰注释。

---

## 第四条：Chrome API 优先原则 (Chrome API First)

**核心：** 充分利用 Chrome 扩展 API，不重复造轮子。

- **4.1 (数据持久化):** 必须使用 chrome.storage.sync 实现数据存储和跨设备同步，不使用 localStorage。
- **4.2 (标签页操作):** 使用 chrome.tabs API 获取当前页面信息。
- **4.3 (插件管理):** 使用 chrome.action API 管理插件图标和弹出窗口。

---

## 第五条：单一职责原则 (Single Responsibility)

**核心：** 每个文件、每个函数都应该只做好一件事。

- **5.1 (文件内聚):** 每个文件保持高度内聚和低耦合。例如，storage.js 只负责数据存储，绝不包含 UI 渲染逻辑。
- **5.2 (组件化):** UI 按功能划分为独立组件（添加按钮、页面列表、编辑表单等），每个组件负责单一职责。

---

## 第六条：可访问性原则 (Accessibility)

**核心：** 插件必须可访问。

- **6.1 (语义化 HTML):** 使用语义化 HTML 结构，便于屏幕阅读器识别。
- **6.2 (键盘导航):** 所有功能必须支持键盘操作。
- **6.3 (ARIA 标签):** 必要时使用适当的 ARIA 标签增强可访问性。

---

## 治理 (Governance)

本宪法具有最高优先级，其效力高于任何 `CLAUDE.md`或单次会话中的指令。任何计划在生成时，都必须首先进行"合宪性审查"。
